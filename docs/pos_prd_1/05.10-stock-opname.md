# 5.10 STOCK OPNAME

## 5.10.1 Dua Jenis SO

| Jenis | Frekuensi | Scope | Tujuan |
|-------|-----------|-------|--------|
| SO Harian | Setiap hari | 20-30 item | Deteksi cepat selisih |
| SO Besar | 1 bulan sekali | Semua SKU | Audit akurasi menyeluruh |

## 5.10.2 Stock Opname Harian

**Target: Selesai dalam 1 jam, akurasi prioritas**

```
STEP 1 — Pilih Produk (Kasir memilih METODE):

Metode A: Auto-Suggest Best Seller (Cycle Counting)
├── Sistem suggest 20-30 produk paling laku HARI INI
└── Kasir checklist mana yang mau di-check

Metode B: Filter "Keluar Hari Ini"
├── Tampilkan SEMUA produk yang terjual hari ini
└── Kasir pilih mana yang mau di-SO

Metode C: Manual Select
└── Kasir pilih bebas dari master produk

STEP 2 — Input Stock Fisik:

Untuk setiap produk terpilih:
├── Tampilkan: "Stock Sistem: 8 Sak, 25 Pcs"
├── Kasir hitung fisik di rak
├── Input: [7] Sak, [30] Pcs
├── Sistem hitung selisih:
│   ├── Sak: -1 (kurang 1)
│   ├── Pcs: +5 (lebih 5)
│   └── Total teoritis selisih: (7×30 + 30) - (8×30 + 25) = 240 - 265 = -25 pcs
└── Indikator warna:
    ├── 🟢 Hijau: pas (0 selisih)
    ├── 🟡 Kuning: lebih (+)
    └── 🔴 Merah: kurang (-)

STEP 3 — Review & Submit:

Tampilkan summary:
├── Total item di-check: 25
├── Pas: 20 item
├── Lebih: 2 item (nilai: +Rp 50.000)
└── Kurang: 3 item (nilai: -Rp 150.000)

Kasir bisa edit jika salah input, lalu "Submit untuk Approval"

STEP 4 — Approval Backoffice:

Manager/Owner review di backoffice:
├── APPROVE → stock adjust, kerugian tercatat
├── REJECT → kembali ke kasir untuk recount
└── REQUEST_RECOUNT → kasir recount item tertentu

STEP 5 — Print Report:

Kasir print hasil SO harian untuk dokumentasi fisik.
```

## 5.10.3 Skip SO Harian

```
Kasir bisa skip SO harian dengan:
├── Input alasan (wajib)
└── Log tercatat

Tracking:
├── Backoffice bisa lihat: "Cabang A skip SO 5x bulan ini"
├── Warning jika skip > 3x dalam seminggu
└── Tidak ada punishment otomatis (keputusan manager)
```

## 5.10.4 Stock Opname Besar (Bulanan)

**Target: Semua SKU, bisa bertahap**

```
PERSIAPAN (Backoffice):
├── Manager buat "SO Besar Periode [Bulan]"
├── Bagi per kategori:
│   ├── Kategori Makanan → Hari 1-3
│   ├── Kategori Obat → Hari 4-6
│   ├── Kategori Aksesoris → Hari 7-9
│   └── dst...
├── Assign tim:
│   ├── 1 orang input (data entry)
│   ├── 1 orang hitung fisik (counter)
│   └── 1 orang supervisi (verifier)
└── Toko TIDAK perlu tutup

EKSEKUSI (di POS):
├── Counter hitung fisik (pakai barcode scanner)
├── Data entry input ke sistem real-time
├── Verifier cek hasil
└── Proses bisa di jam sepi (buka lebih awal atau tutup lebih lama)

APPROVAL:
├── Semua kategori selesai → submit
├── Manager final review
├── Owner final approve
└── Stock adjustment executed

HASIL:
├── Laporan variance per kategori
├── Total kerugian bulan ini
├── Breakdown penyebab:
│   ├── Expired
│   ├── Rusak
│   ├── Hilang
│   └── Salah Input
└── Trend analysis (vs bulan sebelumnya)
```

## 5.10.5 Logika Kerugian

```
Rumus:
Kerugian = (Stock Sistem - Stock Fisik) × Harga Modal (FIFO)

Untuk selisih PLUS (lebih stock):
├── INVESTIGASI dulu (maksimal 2 hari)
├── Kemungkinan:
│   ├── Salah input penjualan (qty lebih)
│   ├── Tertukar dengan produk varian lain
│   └── Barang diterima tapi tidak tercatat
├── Jika memang plus:
│   ├── Adjust stock (tambah)
│   └── Catat sebagai "pengurang kerugian" bulan ini
└── Laporan: "Lebih stock offset kerugian Rp XXX"

Untuk selisih MINUS (kurang stock):
├── INVESTIGASI dulu (maksimal 2 hari)
├── Kategori penyebab (WAJIB input):
│   ├── Expired
│   ├── Rusak
│   ├── Hilang (dicuri/lupa catat)
│   └── Salah Input
├── Adjustment butuh approval OWNER/MANAGER
└── Kerugian tercatat dan direkap akhir bulan
```

## 5.10.6 Rekap Bulanan (Tanggal 30/31)

```

├── Total kerugian: Rp X

├── Breakdown per kategori penyebab

```
