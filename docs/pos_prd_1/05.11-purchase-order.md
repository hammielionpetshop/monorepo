# 5.11 PURCHASE ORDER (PO)

## 5.11.1 Alur Flow PO

```
STEP 1: REQUEST PO (Kasir di POS)
├── Trigger: Auto-suggest (stock < minimum) / Forecast / Manual
├── Kasir buka "Buat PO"
├── Sistem tampilkan suggest produk
├── Kasir tambah produk + pilih supplier + input qty
├── Sistem auto-fill harga beli terakhir
├── (Opsional) Edit harga jika ada info perubahan
└── Submit → Status: DRAFT / PENDING_APPROVAL

STEP 2: FINAL APPROVAL (Owner/Manager di Backoffice)
├── Owner lihat semua PO dari semua cabang
├── Review detail, bisa merge PO (bulk buy)
├── Edit qty/supplier jika perlu
├── APPROVE → Status: APPROVED
├── Sistem generate PO document
└── Kirim ke supplier (email/print/WA)

STEP 3: SUPPLIER KIRIM BARANG
└── Status: IN_TRANSIT → READY_TO_RECEIVE

STEP 4: RECEIVING (Petugas Gudang di POS)
├── Buka PO yang datang
├── Scan barcode barang satu per satu
├── Check: kesesuaian produk, qty, kondisi fisik, expired date
├── Input qty diterima (bisa < PO jika kurang)
├── [PRIORITAS KECEPATAN] Gudang TIDAK WAJIB input harga
│   └── Harga dari "harga beli terakhir" sementara
├── (Opsional) Upload foto + invoice PDF
└── Submit → Status: PARTIALLY_RECEIVED / FULLY_RECEIVED

STEP 5: APPROVAL PENERIMAAN (Backoffice)
├── Finance/Owner verifikasi qty, harga invoice, discrepancy
├── Actions:
│   ├── Kurang qty → follow up supplier (backorder)
│   ├── Harga beda → update + adjust COGS
│   ├── Rusak → return note
│   └── OK → Approve
├── APPROVE:
│   ├── Stock UPDATE per UOM & batch
│   ├── Harga modal update (FIFO)
│   └── Hutang supplier tercatat
└── Status: COMPLETED

STEP 6: PEMBAYARAN SUPPLIER (Backoffice)
├── Finance bayar supplier
├── Input payment (cash/transfer/cek)
├── Hutang supplier berkurang
└── Jika lunas → PO di-archive
```

## 5.11.2 Edit PO

```
PO bisa di-edit HANYA jika:
├── Status: DRAFT / PENDING_APPROVAL / APPROVED (belum received)
├── Yang bisa edit: MANAGER (bukan kasir)
└── Setelah "PARTIALLY_RECEIVED" atau "FULLY_RECEIVED" → tidak bisa edit
```

## 5.11.3 Handling Special Cases

**Case 1: Barang Datang, Invoice Belum**
```
Prioritas: Kecepatan perputaran barang

Flow:
1. Gudang terima barang, input qty
2. Harga = "harga beli terakhir" (sementara)
3. Flag: invoice_received = false
4. Stock tetap masuk, siap dijual
5. Nanti invoice datang → update harga actual
6. Sistem recalculate COGS untuk transaksi setelah penerimaan
```

**Case 2: Qty Kurang dari PO**
```
PO: 100 Sak, Datang: 80 Sak

Action:
├── Input qty diterima: 80 Sak
├── Sistem auto-create Backorder: 20 Sak
├── PO Status: PARTIALLY_RECEIVED
└── Backoffice decide: wait supplier kirim sisa, atau cancel backorder
```

**Case 3: Harga Naik dari PO**
```
PO: Rp 10.000 per Pcs
Invoice: Rp 12.000 per Pcs

Action:
├── Gudang report discrepancy
├── Backoffice review
├── Keputusan:
│   ├── Terima dengan harga baru → update COST PRICE + batch baru @ Rp 12.000
│   └── Reject dan return
└── Logic FIFO: Tetap jual batch lama dulu (Rp 10.000) sampai habis
```

**Case 4: Barang Rusak**
```
Diterima: 100 Pcs, Rusak: 5 Pcs

Action:
├── Input: Received OK = 95, Rusak = 5
├── Foto barang rusak
├── Return note ke supplier
├── Stock update: +95 Pcs only
└── Supplier decide: refund atau replace
```
